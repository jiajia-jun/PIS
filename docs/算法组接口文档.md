# PIS-web 算法组接口文档（拼接脚本）

## 1. 概述

本系统包含两个独立的 Python 脚本，由 Go 后端先后调度：

| 脚本 | 作用 | 执行时机 |
|---|---|---|
| **拼接脚本**（本文档） | 图像拼接，产出全景图 + meta.json | 先执行 |
| **分析脚本**（见《数据分析组接口文档》） | 生成分析图表 PNG | 拼接成功后执行 |

> 本文档仅描述拼接脚本的接口规范。

---

## 2. 调用方式

### 2.1 命令行格式

```
{python解释器} {脚本路径} {任务目录}
```

示例：
```bash
python3 stitch.py E:\xxx\PIS-web\store\uploads\abc-123-def\
```

- python 解释器路径和脚本路径由后端 `config.yaml` 配置
- 最后一个参数是任务目录路径，脚本只收这一个参数

### 2.2 超时限制

Go 后端对脚本调用设置了 **30 秒超时**。超时后进程被强制终止，任务标记为失败。

---

## 3. 输入规范

### 3.1 从哪里读

> **读取路径：`{任务目录}/input/`**

该目录由 Go 后端在上传阶段写入。Python 被调用时，input 目录和 result 目录均已创建。

### 3.2 目录结构

```
{任务目录}/
├── input/                    # ← 【从这里读取】用户上传的原图
│   ├── img_001.jpg
│   ├── img_002.jpg
│   └── ...
└── result/                   # ← 【往这里写入】产出文件
```

### 3.3 文件说明

| 项目 | 说明 |
|---|---|
| 文件格式 | JPG |
| 文件数量 | 不固定（≥1） |
| 命名规则 | `img_001.jpg`、`img_002.jpg`… 按文件名排序 |
| 图片尺寸 | 不保证一致 |

---

## 4. 输出规范

### 4.1 写到哪里

> **输出路径：`{任务目录}/result/`**

### 4.2 必须产出的文件

```
result/
├── meta.json      # 【必须】执行结果元信息
└── result.jpg     # 【必须】拼接后的全景图
```

### 4.3 meta.json 格式

```json
{
    "status": "ok",
    "keypoints": 150,
    "cost_ms": 3200,
    "error": ""
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `status` | string | ✅ | `"ok"` 成功，`"error"` 失败 |
| `keypoints` | int | ✅ | 特征点匹配总数（失败时填 0） |
| `cost_ms` | int64 | ✅ | 执行耗时，单位毫秒（失败时填 0） |
| `error` | string | ✅ | 成功为空，失败写原因 |

### 4.4 result.jpg 规格

| 项目 | 说明 |
|---|---|
| 格式 | JPEG |
| 内容 | 拼接完成的全景图 |
| 质量 | 建议 85–95 |

---

## 5. 错误处理

失败时仍需写入 `meta.json`：

```json
{
    "status": "error",
    "keypoints": 0,
    "cost_ms": 0,
    "error": "输入图像重叠区域不足，无法拼接"
}
```

- `status` 设为 `"error"`
- `error` 写清楚原因，支持中文
- 失败时不生成 `result.jpg`

---

## 6. 完整流程

```
Go 后端                                      拼接脚本
───────                                      ────────
1. 用户上传图片
2. 存入 {任务目录}/input/
3. 执行 python3 stitch.py {任务目录}  ──→   4. 读取 input/ 中的原图
                                             5. 执行拼接算法
                                             6. 写入 result/result.jpg
                                             7. 写入 result/meta.json
8. 读取 meta.json                    ←──
9. 更新数据库
10. 拼接成功后，后端继续调用分析脚本
11. 10 分钟后清理 input/ 目录
```

---

## 7. 调试说明

- 后端默认使用 Mock 引擎（不调脚本）。开发时将 `config.yaml` 中 `engine.stitch_type` 改为 `python` 才会真实调用
- 任务目录下的文件保留 10 分钟才清理
- 执行日志位于 `logs/task.log`
