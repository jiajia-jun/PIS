# PIS-web 后端设计文档

## 1. 项目概述

**PIS-web**（Panoramic Image Stitching）是一个全景图像拼接系统的 Web 后端。用户上传多张有重叠区域的图片，后端异步处理后返回拼接结果，并支持历史记录查看与数据分析展示。

---

## 2. 技术栈

| 组件 | 技术选型 |
|---|---|
| 语言 | Go 1.25.5 |
| Web 框架 | Gin |
| ORM | GORM |
| 数据库 | MySQL |
| 配置 | config.yaml（gopkg.in/yaml.v3） |
| 图片处理 | imaging 库（缩略图） |
| 并发 | Goroutine + Channel |
| 跨语言调用 | os/exec 调用 Python 脚本 |

## 4. 架构设计

### 4.1 分层架构

```
┌─────────────────────────────────────┐
│            Handler 层               │  ← Gin HTTP 请求/响应
├─────────────────────────────────────┤
│           Middleware 层             │  ← 文件大小限制、日志
├─────────────────────────────────────┤
│           Service 层                │  ← 业务逻辑编排
├──────────┬──────────┬───────────────┤
│  Engine  │  Worker  │     Tool      │  ← 核心模块
│  策略模式 │  并发调度 │   图片处理    │
├──────────┴──────────┴───────────────┤
│            Model 层                 │  ← GORM 数据模型
├─────────────────────────────────────┤
│          Database 层                │  ← MySQL 连接管理
└─────────────────────────────────────┘
```

### 4.2 请求数据流

```
POST /api/upload (多图)
    │
    ▼
Handler: 验证文件大小 → 保存图片到 store/uploads/{task_id}/input/
    │
    ▼
Service: 创建 Task 记录 → 构建 Job → 塞入 Channel → 返回 task_id
    │
    ▼
Worker: 从 Channel 取 Job → Engine.Run() → Service.HandleResult()
    │
    ▼
Service.HandleResult(): 解析 meta.json → 更新 DB → 注册 AfterFunc 清理

GET /api/task/:task_id (轮询)
    │
    ▼
Handler: 查询 DB → 返回任务状态 + 结果 URL（完成时）
```

### 5.2 Model（数据模型）

```go
type Task struct {
    ID         string    `gorm:"primaryKey;size:36"`
    Status     string    `gorm:"size:20;index"`
    CostMs     int64
    Keypoints  int
    ErrorMsg   string    `gorm:"size:500"`

```

**状态枚举**：
| 状态 | 说明 |
|---|---|
| `pending` | 已入队，等待处理 |
| `processing` | Worker 正在处理 |
| `completed` | 拼接成功 |
| `failed` | 拼接失败 |

**建表方式**：AutoMigrate。

```go
// 数据库初始化

```

### 5.3 Engine（算法引擎 — 策略模式）

系统有两个独立的算法引擎，通过 `config.yaml` 各自独立切换 mock/python：

**拼接引擎** 和 **分析引擎** 均实现统一的 `AlgorithmEngine` 接口：

```go
// 引擎统一接口
type AlgorithmEngine interface {
    Run(ctx context.Context, taskDir string) (*Meta, error)
    Name() string
}

// 返回结果
type Meta struct {
    Status    string `json:"status"`
    Keypoints int    `json:"keypoints"`
    CostMs    int64  `json:"cost_ms"`
    Error     string `json:"error"`
}
```

**拼接引擎实现：**

| 实现 | Name | 行为                                                        |
|---|---|-----------------------------------------------------------|
| `MockEngine` | `"mock"` | 取 input 目录第一张图复制为 result.jpg，sleep 1s，返回模拟 Meta           |
| `PythonEngine` | `"python"` | `os/exec`: `{python_path} {script_path} {taskDir}`，30s 超时 |
| `CppEngine` | `"cpp"` | 空实现，以供拓展开发                                                |

**分析引擎实现：**

| 实现 | Name | 行为 |
|---|---|---|
| `MockAnalysisEngine` | `"mock-analysis"` | sleep 0.5s，在 `store/analysis/{task_id}/` 生成占位 PNG |
| `PythonAnalysisEngine` | `"python-analysis"` | `os/exec`: `{python_path} {script_path} {taskDir}`，30s 超时 |

**执行顺序**：Worker 先执行拼接引擎 → 拼接成功 → 执行分析引擎 → 分析失败不影响拼接结果

### 5.4 Worker Pool（异步任务队列）

```go
type Job struct {
    TaskID         string
    TaskDir        string                  // store/uploads/{task_id}
    StitchEngine   engine.AlgorithmEngine  // 拼接引擎
    AnalysisEngine engine.AlgorithmEngine  // 分析引擎
}

type Pool struct {
    jobs    chan Job       // 有缓冲，容量 3
    service *service.TaskService
    timeout time.Duration
    logger  *zap.Logger
}
```

**Worker 执行流程（先拼接后分析）**：
```go
func (p *Pool) processJob(job Job) {
    // 1. 更新状态为 processing
    // 2. 执行拼接引擎（带超时）
    meta, stitchErr := job.StitchEngine.Run(stitchCtx, job.TaskDir)
    if stitchErr != nil {
        // 拼接失败，跳过分析，直接记录失败
        p.service.HandleResult(job.TaskID, job.TaskDir, meta, stitchErr)
        return
    }
    // 3. 拼接成功，执行分析引擎（带超时）
    _, analysisErr := job.AnalysisEngine.Run(analysisCtx, job.TaskDir)
    // 4. 分析失败仅记录日志，不影响拼接结果
    // 5. 交给 Service 处理结果
    p.service.HandleResult(job.TaskID, job.TaskDir, meta, nil)
}
```

**关键约束**：
- Worker 数量固定为 3
- Channel 容量为 3（满时上传接口阻塞，自然背压）
- 每个 Python 调用超时 30 秒
- 超时或失败不崩溃，记录 ErrorMsg 并标记 failed

### 5.6 Handler（HTTP 处理器）

| 方法 | 路径 | Handler | 说明 |
|---|---|---|---|
| POST | `/api/upload` | `Upload` | 解析 multipart 表单，校验数量，保存文件，创建任务，返回 task_id |
| GET | `/api/task/:task_id` | `GetTask` | 查 DB 返回状态。完成时附带 result_url 和 analysis_urls |
| GET | `/api/history` | `GetHistory` | Query 参数 `page`（默认1）, `size`（默认10） |
| GET | `/api/result/:task_id` | `GetResult` | 返回 result.jpg 原图，`Content-Type: image/jpeg` |
| GET | `/api/thumbnail/:task_id` | `GetThumbnail` | 从 result.jpg 内存缩放（800px），返回 JPEG 流，不落盘 |
| GET | `/api/analysis/:task_id/:filename` | `GetAnalysis` | 返回分析图表 PNG 二进制流 |

**Upload Handler 逻辑**：
```go
func (h *UploadHandler) Upload(c *gin.Context) {
    form, _ := c.MultipartForm()
    files := form.File["images"]
    // 1. 校验文件数量 > 0
    // 2. 生成 taskID (UUID)
    // 3. 创建目录 store/uploads/{taskID}/input/ 和 result/
    // 4. 保存所有文件到 input/
    // 5. 生成缩略图
    // 6. 创建 DB 记录（status=pending）
    // 7. 构建 Job 塞入 Pool.Channel
    // 8. 返回 {"task_id": taskID}
}
```

### 5.8 日志系统

#### 5.8.1 日志架构

```
logs/
├── access.log          # HTTP 请求日志
└── task.log            # 任务处理日志
```

两条日志流**完全独立**，互不干扰：

| 日志 | 写入位置 | 写入模块 | 触发时机 |
|---|---|---|---|
| 请求日志 | `logs/access.log` | `middleware/logger.go` | 每个 HTTP 请求结束时 |
| 任务日志 | `logs/task.log` | `service/task_service.go` | 任务处理完成时（HandleResult） |

#### 5.8.2 请求日志（access.log）

**格式**：
```
[2026-07-06 14:30:22] POST /api/upload 200 45ms
[2026-07-06 14:30:25] GET /api/task/abc-123 200 2ms
[2026-07-06 14:30:28] GET /api/history?page=1&size=10 200 5ms
```

**字段**：`[时间戳] HTTP方法 路径 状态码 耗时`

**实现方式**：Gin 中间件 + `go.uber.org/zap`。中间件在请求结束时用 zap 的 `Info` 写入字段。日志格式使用 ConsoleEncoder（人眼可读）。

#### 5.8.3 任务日志（task.log）

**格式**：
```
[2026-07-06 14:30:22] task_id=abc-123 status=completed cost_ms=3200 keypoints=150 images=5
[2026-07-06 14:31:05] task_id=def-456 status=failed error="context deadline exceeded" images=3
[2026-07-06 14:32:10] task_id=ghi-789 status=completed cost_ms=5100 keypoints=230 images=8
```

**字段**：`[时间戳] task_id=xxx status=xxx [cost_ms=xxx] [keypoints=xxx] [error=xxx] images=xxx`

- `cost_ms` 和 `keypoints` 仅在 `status=completed` 时输出
- `error` 仅在 `status=failed` 时输出
- `images` 始终输出

**实现方式**：`TaskService` 内部持有一个独立的 `*zap.Logger`（写入 `logs/task.log`），`HandleResult` 根据任务状态写入对应字段。

#### 5.8.4 日志初始化

`main()` 启动时创建两个独立的 `zap.Logger` 实例：

```go
func InitAccessLogger() *zap.Logger {
    cfg := zap.NewProductionConfig()
    cfg.Encoding = "console"
    cfg.EncoderConfig.EncodeTime = zapcore.TimeEncoderOfLayout("2006-01-02 15:04:05")
    cfg.OutputPaths = []string{"logs/access.log"}
    logger, _ := cfg.Build()
    return logger
}

func InitTaskLogger() *zap.Logger {
    cfg := zap.NewProductionConfig()
    cfg.Encoding = "console"
    cfg.EncoderConfig.EncodeTime = zapcore.TimeEncoderOfLayout("2006-01-02 15:04:05")
    cfg.OutputPaths = []string{"logs/task.log"}
    logger, _ := cfg.Build()
    return logger
}
```

- 两个 `*zap.Logger` 完全独立，各自写各自的文件
- 均使用 ConsoleEncoder
- `main()` 退出时 `defer logger.Sync()` 确保刷盘

### 5.9 Tool（图片处理）

```go
// 从 reader 读取图片，缩放到指定宽度，以 JPEG 写入 writer（纯内存操作）
func ResizeToJPEG(src io.Reader, dst io.Writer, width int) error
```

使用 `github.com/disintegration/imaging` 库，纯内存操作不落盘。

**缩略图策略**：
- 上传时不生成缩略图
- 前端先请求 `GET /api/thumbnail/:task_id` 获取 800px JPEG 流（约 5KB，快速展示）
- 同时后台请求 `GET /api/result/:task_id` 加载原图，加载完毕替换

**文件存放结构**：
```
{task_id}/
├── input/               # 原始上传文件（10分钟后清理）
│   ├── img_001.jpg
│   ├── img_002.jpg
│   └── ...
└── result/              # Python 产出（不清理）
    ├── result.jpg
    └── meta.json
```
