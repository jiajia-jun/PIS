## 1. 系统总体架构

PIS-web（Panoramic Image Stitching Web）是一个基于 B/S 架构的全景图像拼接系统。用户通过 Web 浏览器上传多张具有重叠区域的图像，后端异步调用 Python 算法脚本完成拼接与质量评估，最终将全景图及分析图表返回前端展示。

## 2. 模块划分与职责

### 2.1 模块总览表

| 模块 | 包/目录 | 层级 | 核心职责 |
|------|---------|------|---------|
| **入口** | `main.go` | 启动 | 调用 `app.New()` + `Run()` |
| **应用组装** | `internal/app/` | 编排 | 依赖注入：创建并关联所有组件 |
| **配置** | `internal/config/` | 基础 | 解析 `config.yaml` + 环境变量 |
| **数据库** | `internal/database/` | 持久化 | GORM 初始化、AutoMigrate |
| **数据模型** | `internal/model/` | 领域 | Task 实体 + 状态常量 |
| **算法引擎** | `internal/engine/` | 算法 | Strategy Pattern 接口 + 3 种实现 |
| **服务层** | `internal/service/` | 业务 | 任务 CRUD、结果处理、历史淘汰 |
| **Worker 池** | `internal/worker/` | 并发 | 固定协程数任务调度 + 缩略图生成 |
| **HTTP 处理** | `internal/handler/` | 接口 | Gin 路由处理，7 个 Handler |
| **中间件** | `internal/middleware/` | 横切 | 请求体限制、访问日志 |
| **图像工具** | `internal/tool/` | 工具 | 图片缩放（Lanczos → JPEG） |
| **Python 脚本** | `scripts/` | 算法 | stitch.py（拼接）+ eval_pipeline.py（分析） |
| **前端** | `frontend/` | 展示 | Vue 3 SPA，上传/详情/历史三个页面 |
| **文档** | `docs/` | 文档 | 接口规范、数据交互规范、错误汇总 |

### 2.2 各模块详细说明

#### 2.2.1 入口模块 (`main.go`)

```
输入: 无
输出: 进程启动/退出
依赖: internal/app
```

职责极简，仅负责创建 `App` 实例并启动。所有初始化逻辑封装在 `app` 包中。

#### 2.2.2 应用组装模块 (`internal/app/app.go`)

```
输入: config.yaml 文件路径
输出: 组装完成的 App 实例（持有所有顶层依赖）
依赖: config, database, engine, handler, middleware, service, worker
```

按顺序执行初始化：
1. `config.Load()` — 加载配置
2. `initLoggers()` — 创建访问日志 + 任务日志
3. `initDatabase()` — 连接 MySQL + 自动建表
4. `initEngines()` — 工厂方法创建 Stitch/Analysis 引擎
5. `initService()` — 创建 TaskService
6. `initWorkerPool()` — 创建并启动 Worker Pool
7. `initRouter()` — 注册 API 路由 + 静态文件服务

#### 2.2.5 数据模型模块 (`internal/model/task.go`)

```
Task 实体字段:
  ID         string    (UUID, 主键)
  Status     string    (pending|processing|completed|failed)
  CostMs     int64     (拼接耗时，毫秒)
  Keypoints  int       (特征点总数)
  ErrorMsg   string    (错误信息)

  ResultPath string    (结果图磁盘路径)
  CreatedAt  time.Time
  UpdatedAt  time.Time
```

状态机：`pending → processing → completed|failed`

#### 2.2.6 算法引擎模块 (`internal/engine/`)

**核心接口：**

```go
type AlgorithmEngine interface {
    Run(ctx context.Context, inputDir string) (*Meta, error)
    Name() string
}

type Meta struct {
    Status    string `json:"status"`    // "ok" | "error"
    Keypoints int    `json:"keypoints"`
    CostMs    int64  `json:"cost_ms"`
    Error     string `json:"error"`
}
```

**Stitch 引擎（拼接）三种实现：**

| 引擎 | 文件 | 说明 |
|------|------|------|
| `MockEngine` | `mock.go` | 模拟 1 秒延迟，复制首张图为结果，写假 meta.json |
| `PythonEngine` | `python.go` | `os/exec` 调用 `stitch.py {inputDir}`，读 `meta.json` 返回 |
| `CppEngine` | `cpp.go` | 占位实现，返回"待开发"错误 |

**Analysis 引擎（分析）三种实现：**

| 引擎 | 文件 | 说明 |
|------|------|------|
| `MockAnalysisEngine` | `analysis_mock.go` | 生成 1 像素占位 PNG，验证链路 |
| `PythonAnalysisEngine` | `analysis_python.go` | `os/exec` 调用 `eval_pipeline.py {taskDir} {analysisDir}` |
| （无 C++ 版本） | — | 分析引擎无需 C++ 实现 |

**引擎选择策略（工厂方法，位于 `app.go`）：**

```
config.Engine.StitchType ──► switch ──┬── "python" → PythonEngine
                                      ├── "cpp"    → CppEngine
                                      └── default  → MockEngine

config.Engine.AnalysisType ──► switch ─┬── "python" → PythonAnalysisEngine
                                       └── default  → MockAnalysisEngine
```

拼接和分析引擎**独立可切换**，例如开发时可 `stitch=python, analysis=mock`。

#### 2.2.7 服务层 (`internal/service/`)

| 文件 | 方法 | 输入 | 输出 |
|------|------|------|------|
| `task_service.go` | `CreateTask(imageCount)` | 图片数量 | *Task |
| | `GetTask(taskID)` | UUID | *Task |
| | `UpdateTask(task)` | *Task | error |
| | `GetHistory(page, size)` | 分页参数 | []Task, total |
| | `HandleResult(taskID, dir, meta, err)` | 引擎结果 | 无（更新 DB + 写日志） |
| | `writeTaskLog(task)` | *Task | 写 `logs/task.log` |
| `task_eviction.go` | `enforceHistoryLimit()` | 无 | 超出 30 条时删旧记录+文件 |

**`HandleResult` 处理逻辑：**
1. 若 `runErr != nil`（Go 层超时/Python 崩溃）→ 标记 `failed`
2. 读 `meta.json` 解析失败 → 标记 `failed`
3. `meta.Status == "error"` → 标记 `failed`，透传 `meta.Error`
4. `meta.Status == "ok"` → 标记 `completed`，填充 `CostMs`、`Keypoints`、`ResultPath`

#### 2.2.8 Worker Pool 模块 (`internal/worker/`)

```
配置参数:
  PoolSize:        20 个 goroutine
  TimeoutSeconds:  30 秒（单个 Python 进程超时）
  Channel 容量:    PoolSize（有缓冲）

Job 结构体:
  TaskID          string
  TaskDir         string   // store/uploads/{task_id}
  StitchEngine    AlgorithmEngine
  AnalysisEngine  AlgorithmEngine
```

**任务处理流程（`processJob`）：**

```
┌──────────────────────────────────────────────────────────────┐
│                     Worker.processJob(job)                    │
│                                                              │
│  1. 更新 Task.Status → "processing"                          │
│                         │                                    │
│  2. StitchEngine.Run(ctx, inputDir)  ← 30s 超时              │
│                         │                                    │
│          ┌──────────────┼──────────────┐                     │
│          ▼              ▼              ▼                     │
│       成功           失败/超时      meta.Status="error"       │
│          │              └──────→ HandleResult (failed)        │
│          ▼                                                   │
│  3. AnalysisEngine.Run(ctx, taskDir)  ← 30s 超时             │
│          │                                                   │
│          ├── 成功 → 继续                                     │
│          └── 失败 → 记 warning 日志，不影响拼接结果           │
│                         │                                    │
│  4. generateThumbnail(result.jpg → thumb.jpg)                │
│          │                                                   │
│          ├── 成功 → 继续                                     │
│          └── 失败 → 记 warning 日志，不影响结果               │
│                         │                                    │
│  5. HandleResult(taskID, taskDir, meta, nil)                  │
│     → 读 meta.json → 标记 completed → 写任务日志             │
└──────────────────────────────────────────────────────────────┘
```

**关键设计决策：**
- 分析失败**不阻塞**拼接成功：拼接成功后即使分析出错，任务仍标记 `completed`
- 缩略图失败**不阻塞**：仅记日志，不影响任务完成状态
- 文件清理：`HandleResult` 执行完毕后触发 `enforceHistoryLimit()`，最多保留 30 条完成/失败记录

#### 2.2.12 Python 脚本 (`scripts/`)

**stitch.py — 拼接脚本：**

```
输入:  CLI 参数 {任务目录}
      读取 {任务目录}/input/*.jpg
输出:  {任务目录}/result/result.jpg    拼接全景图
       {任务目录}/result/meta.json     执行元信息
       {任务目录}/result/result_info.json  匹配统计（分析组消费）
       {任务目录}/result/H_list.npy    单应矩阵列表（分析组消费）
       {任务目录}/result/inliers_list.pkl  内点坐标（分析组消费）

关键算法:
  - SIFT 特征检测（不可用时回退 ORB），一次性提取全部图片描述子并缓存复用
  - BFMatcher + knnMatch + Lowe's ratio test（阈值 0.7）
  - RANSAC 单应矩阵估计（DLT 最小解算 + 5px 内点阈值 + 最小二乘精化）
  - OpenCV Stitcher（球面 PANORAMA → 平面 SCAN 自动回退，置信度 0.6）
  - setWaveCorrection(False) — 关闭波浪校正，避免非水平排列畸变
  - 最长边限制 4000px + 黑边自动裁剪
  - 预处理缩放最长边 2000px
  - 输入上限 MAX_IMAGES=20，防止内存溢出
  - 磁盘空间不足时区分"磁盘满"与"权限/路径问题"

产出文件:
  - result.jpg              拼接全景图
  - meta.json                执行元信息（status/keypoints/cost_ms/error）
  - result_info.json         逐对匹配统计（pairs/total_time）
  - H_list.npy               单应矩阵 (n,3,3)
  - inliers_list.pkl         内点坐标列表
```

**eval_pipeline.py — 分析脚本：**

```
输入:  CLI 参数 {任务目录} {分析输出目录}
      读取 stitch.py 产出的中间数据（H_list.npy, inliers_list.pkl, result_info.json）
      读取 result.jpg 和 input/*.jpg
输出:  {分析输出目录}/eval_result.json   精简评估摘要（6 项指标）
       {分析输出目录}/full_metrics.json  分组详细数据（6 类 × 多字段）
       {分析输出目录}/metrics_dashboard.png  仪表盘总览
       {分析输出目录}/quality_radar.png      雷达图
       {分析输出目录}/gauge_*.png × 6        各指标独立大图
       {分析输出目录}/metrics_table.png      指标明细数据表

三步处理:
  A. 入参桥接 → stitch.py 新格式转为分析核心旧格式
  B. 调用 eval_core.run_production_analysis() 核心分析
  C. 出参转换 → 生成中文图表 + eval_result.json + full_metrics.json
```

### 3.2 Go ↔ Python IPC 协议

```

  通过 CLI 参数传递路径:
    stitch.py    → 1个参数: {任务目录}         (含 input/ + result/)
    eval_pipeline.py  → 2个参数: {任务目录} {分析输出目录}

  通过约定文件:
    result/meta.json  → Go 读取判断成功/失败
    result/result.jpg → Go 直接服务为静态资源

Python → 分析组:
  stitch.py 产出中间数据 → eval_pipeline.py 读取:
    result/H_list.npy
    result/inliers_list.pkl
    result/result_info.json
    result/result.jpg

  eval_pipeline.py 产出 → Go 通过 /api/analysis/ 暴露:
    *.png  (图表)
    eval_result.json  (摘要表格)
    full_metrics.json (详细数据)

优势:
  - 解耦: 三方（Go/Python/分析组）通过文件约定通信，独立开发

  - 无序列化开销: 大文件（图片）以二进制存盘，不经过 base64
```

## 6. 关键设计模式与技术决策

### 6.1 策略模式（Strategy Pattern）

**问题：** 算法引擎需要在多种实现间切换（Mock 开发测试、Python 生产、C++ 未来扩展），且切换不应影响上层代码。

**方案：** 定义 `AlgorithmEngine` 接口，`app.go` 中的工厂方法根据 `config.yaml` 选择具体实现。Handler 和 Worker 仅依赖接口，不感知底层引擎类型。

```
config.yaml                  app.go 工厂                 调用方
─────────────              ┌──────────────┐           ┌──────────┐
stitch_type: python ──────►│ newStitchEngine│ ──────►│  Handler  │
analysis_type: mock ──────►│ newAnalysis   │ ──────►│  Worker   │
                           └──────────────┘           └──────────┘
                                                          │
                                                   依赖 AlgorithmEngine
                                                   接口，不依赖具体实现
```

### 6.2 Worker Pool 模式

**问题：** Python 拼接进程是 CPU 密集型任务，若每个请求同步等待会阻塞 HTTP 线程；若无限制并发则可能 OOM。

**方案：** 固定 20 个 goroutine 从有缓冲 Channel 取 Job，控制并发数。上传接口异步提交 Job 后立即返回 `task_id`，前端轮询获取结果。

### 6.3 文件系统 IPC

**问题：** Go 与 Python 之间需要传递大图片数据和结构化元信息。

**方案：** 大文件（图片）通过文件系统传递，结构化数据（meta.json）以 JSON 文件约定格式。避免 base64 编码开销和 stdin/stdout 解析复杂度。

### 6.4 优雅关闭（Graceful Shutdown）

`App.Run()` 监听 `SIGINT/SIGTERM` 信号，收到后调用 `context.Cancel()` 通知 Worker Pool 停止接收新任务，等待 500ms 后同步日志缓冲区并退出。

### 6.5 前端渐进加载

- **缩略图占位：** 任务完成时预生成 800px 缩略图，前端先显示模糊占位图，`el-image` 的 `placeholder` 槽渲染缩略图，原图加载完成后替换
- **轮询机制：** 前端每 1 秒轮询 `GET /api/task/{id}`，状态变为 `completed` 或 `failed` 后停止轮询
- **分析数据按需加载：** `eval_result.json` 和 `full_metrics.json` 仅在任务完成后异步 fetch，失败不影响主结果展示

### 6.6 历史淘汰机制

最多保留 30 条 `completed` 或 `failed` 记录。每次 `HandleResult` 后检查，超出部分按创建时间升序删除 — 同时清理磁盘文件（`store/uploads/{id}` 和 `store/analysis/{id}`）和数据库记录。

## 7. 数据库设计

### 7.1 Task 表结构

```sql
CREATE TABLE tasks (
    id          VARCHAR(36)  PRIMARY KEY,       -- UUID
    status      VARCHAR(20)  DEFAULT 'pending', -- pending|processing|completed|failed
    cost_ms     BIGINT       DEFAULT 0,         -- 拼接耗时(毫秒)
    keypoints   INT          DEFAULT 0,         -- 特征点总数
    error_msg   VARCHAR(500) DEFAULT '',        -- 错误信息
    image_count INT          DEFAULT 0,         -- 上传图片数
    result_path VARCHAR(255) DEFAULT '',        -- 结果图路径
    created_at  DATETIME,                       -- 创建时间
    updated_at  DATETIME,                       -- 更新时间
    INDEX idx_status (status)
);
```

### 7.2 状态流转

```
  pending ──► processing ──► completed
                │
                └──────────► failed
```

- `pending`：任务已创建，等待 Worker 调度
- `processing`：Worker 正在执行拼接
- `completed`：拼接成功，结果可访问
- `failed`：拼接失败（超时/算法错误/Python 异常）

```
    │       ├── 元信息条 (耗时/特征点/图片数)
    │       ├── 结果图区 (缩略图占位 → 原图替换)
    │       ├── 评估摘要区 (eval_result.json 表格 + 雷达图)
    │       ├── 详细数据面板 (折叠 → el-tabs 分6组)
    │       ├── 原始图片网格
    │       ├── 分析图表网格 (PNG, 去重雷达图)
    │       └── 分析数据表 (JSON → el-table)
    │
    ├── /history → HistoryView.vue
    │   └── el-table (分页) → 点击行 → router.push(/task/{id})
    │
    └── /about → AboutView.vue
        └── 项目介绍 / 技术栈 / 团队成员 / 联系方式
```

## 10. 安全措施

| 措施 | 位置 | 说明 |
|------|------|------|
| 请求体大小限制 | `middleware/limit.go` | 200MB 硬限制，超限返回 413 |
| 路径穿越防护 | `handler/input.go` | `filepath.Base(filename) != filename` 拒绝含 `..` 的路径 |
| MySQL 连接串 | 环境变量 `DATABASE_PIS` | 敏感信息不入 yaml、不提交 git |
| 优雅关闭 | `app/app.go` | 收到信号后停止接收新任务，刷新日志缓冲 |
| 上传文件隔离 | `store/uploads/{uuid}/` | 每个任务独立目录，UUID 不可猜测 |
| Python 脚本超时 | `context.WithTimeout(30s)` | 防止失控进程占用资源 |
