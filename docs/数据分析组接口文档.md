# PIS-web 数据分析组接口文档（分析脚本 `analysis.py`）

## 1. 定位

| 脚本 | 路径 | 参数个数 | 执行时机 |
|---|---|---|---|
| stitch.py（见算法组文档） | `./scripts/stitch.py` | 1 个 | 先执行 |
| **analysis.py**（本文档） | `./scripts/analysis.py` | **2 个** | 拼接成功后执行 |

> 拼接失败的任务**不会**调用分析脚本。

---

## 2. 调用方式

```bash
{python解释器} ./scripts/analysis.py {任务目录} {分析输出目录}
```

示例：
```bash
python3 ./scripts/analysis.py ./store/uploads/abc-123-def/ ./store/analysis/abc-123-def/
```

| 参数 | 含义 | 说明 |
|---|---|---|
| 第1个 | 任务目录 | 从此目录读取拼接结果 |
| 第2个 | 分析输出目录 | 往里面写 PNG 图表 |

- **收 2 个参数**
- 超时限制：**30 秒**
- 分析失败**不影响拼接结果**，拼接仍标记为 completed

---

## 3. 目录结构

脚本被调用时，任务目录结构如下（拼接已完成）：

```
{任务目录}/                         # ← CLI 第1个参数
├── input/                          # ← 【读】原始图片
│   ├── img_001.jpg
│   └── ...
└── result/                         # 拼接产出
    ├── meta.json                   # （Go 后端用，分析脚本无需读取）
    └── result.jpg                  # ← 【读】拼接全景图

{分析输出目录}/                      # ← CLI 第2个参数（Go 后端已创建，直接写入）
```

---

## 4. 读：由算法组独立传输数据，独立生成分析数据

> 分析脚本**不读 meta.json**。meta.json 是 Go 后端用来给前端展示历史记录（状态、耗时、特征点数）的，与分析脚本无关。
>
> 数据分析组请自行与算法组协调，进行信息的读取

## 🚨🚨🚨5. 写：产出 PNG 图表 + JSON 表格到 `{分析输出目录}`

### 5.1 `{分析输出目录}` 解释

`{分析输出目录}` 就是 CLI 的 **第2个参数**，Go 后端直接传过来的完整路径，例如：

```
./store/analysis/abc-123-def/
```

**Go 后端已经创建好该目录**，脚本拿到手就能直接往里写 PNG，**不需要自己建目录**。

### 5.2 往里面写 PNG

Go 后端在调用脚本前已经执行了 `os.MkdirAll`，所以脚本**不需要自己建目录**。

`sys.argv[2]` 拿到的就是一个已存在的目录路径，例如 `./store/analysis/abc-123-def/`，直接往里 `plt.savefig` 即可：
- 下面这段代码供数据组参考
```python
import sys, os
import matplotlib.pyplot as plt

out_dir = sys.argv[2]    # Go 已创建，例如 ./store/analysis/abc-123-def/

# 直接写入，无需 os.makedirs
plt.plot(...)
plt.savefig(os.path.join(out_dir, "cost_trend.png"))
```

> 对比：算法组的 `result/` 目录也是 Go 预先建好的，脚本直接用。两组逻辑完全一致——Go 管目录，脚本管内容。

### 5.3 产出规范

| 项目 | 说明                                                    |
|---|-------------------------------------------------------|
| 格式 | **PNG**（图表） + **JSON**（表格）                            |
| PNG 数量 | 不固定，按分析维度决定                                           |
| JSON 数量 | 固定 1 个，文件名固定为 **`eval_result.json`**                   |
| PNG 命名 | 建议语义化英文，不含中文。如 `cost_trend.png`、`keypoints_chart.png` |

最终产出示例：
```
./store/analysis/abc-123-def/      # ← 就是 sys.argv[2]，Go 后端已创建
├── cost_trend.png
├── keypoints_chart.png
└── eval_result.json               # ← 新增：评估数据表格
```

###🚨🚨 5.4 `eval_result.json` 格式规范

> ⚠️ **产出的文件名固定为 `eval_result.json`**，不可修改。后端按文件名识别表格数据。

```json
{
  "title": "全景拼接评估结果",
  "headers": ["用例名称", "状态", "总匹配对数", "RANSAC内点数", "内点率"...],
  "rows": [
    ["demo2", "成功", 184, 136, 0.7391...],
    ["demo3", "成功", 161, 145, 0.9006...]
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `title` | string | ✅ | 表格标题，前端展示用 |
| `headers` | string[] | ✅ | 列名数组，顺序与 rows 中每行数据一一对应 |
| `rows` | any[][] | ✅ | 数据行，每行是一个数组，元素顺序与 headers 一致。数值直接传 number 类型 |
这些字段里面的内容按照数据组具体所提供的数据进行修改
> 每个任务独立一份 `eval_result.json`，只包含该次拼接的评估数据（通常 1 行）。如果分析脚本需要对多个用例做汇总评估，另行与后端组协商。

## 6. 前端如何访问

分析图表和表格通过 API 暴露：

```
GET /api/analysis/{task_id}/{filename}
```

历史记录中每条已完成任务会携带 `chart_urls` 和 `table_urls`：

```json
{
    "task_id": "abc-123-def",
    "status": "completed",
    "chart_urls": [
        "/api/analysis/abc-123-def/cost_trend.png",
        "/api/analysis/abc-123-def/keypoints_chart.png"
    ],
    "table_urls": [
        "/api/analysis/abc-123-def/eval_result.json"
    ]
}
```

- `chart_urls` — 所有 PNG 图表，前端用 `<img>` 渲染
- `table_urls` — 所有 JSON 表格（目前固定为 `eval_result.json`），前端用表格组件渲染

> 文件名需与脚本实际产出文件名一致，请与前后端沟通确定。

---

## 7. 容错

- 分析脚本执行失败 → 拼接依然标记为 completed，仅在后端日志 `logs/task.log` 中记录 warning
- 图表不存在时前端对应位置留空

---

## 8. 速查总表

| | 说明                         |
|---|----------------------------|
| **参数** | 2 个：任务目录 + 分析输出目录          |
|**读什么**| 与算法组单独沟通，务必让算法组告知后端组数据传输流程 |
| **写什么** | PNG 图表（数量不固定）              |
| **写到哪里** | `{分析输出目录}`（Go 后端已创建，脚本直接写） |
| **失败行为** | 不影响拼接结果，后端仅记日志             |
| **超时** | 30 秒                       |
