## 3. 差异根因

### 3.1 H 矩阵方向约定相反（一级差异）

**验证方法**: 对同一组 inliers，分别计算 `H@src → dst`（正确方向）和 `H@dst → src`（反方向）的重投影误差。

| 测试 | 数据组 H | PIS-web H |
|---|---|---|
| `H@src → dst` (OpenCV 标准) | **1566 px** ✗ | **~3 px** ✓ |
| `H@dst → src` (非标准反向) | **1.3 px** ✓ | 不做测试 |

**结论**: 数据组的 `H.npy` 满足 `H·dst ≈ src`，即 H 映射 dst→src。这与 OpenCV `findHomography(src, dst)` 的标准约定（H 映射 src→dst）相反。

**来源**: 数据组的匹配代码可能使用了 `findHomography(dst_pts, src_pts)` 或等价地交换了 src/dst 角色。

**连锁影响**: 修复前的 test5.py 恰好使用错误方向（`H@dst vs src`），与数据组 H 内恰自洽。修复后 test5.py 使用正确方向（`H@src vs dst`），加载数据组 H 会产生 1566px 的荒谬 RMSE。

### 3.2 匹配点集完全不同（二级差异）

同一对 8MP 图片，两条管线提取出的特征对应关系是两套完全不同的点集：

```
数据组 inliers 坐标范围:
  src x: [791, 1172]  y: [9, 741]
  dst x: [ 47,  429]  y: [22, 757]
  → x 方向位移均 ~748px，几乎纯横向平移，43 对点一致性极高

PIS-web inliers 坐标范围:
  → 882 对点，分布在更广泛的坐标区域
  → 同样识别出横向平移，但 H 矩阵的旋转和缩放分量不同
```

**原因**: 两条管线使用不同的特征检测器、不同的筛选策略，导致输入 RANSAC 的匹配对集合完全不同。RANSAC 在不同点集上收敛到不同的 H。

| 可能的差异点 | 数据组（推测） | PIS-web |
|---|---|---|
| 特征检测器 | 可能非 SIFT（点数极少） | `cv2.SIFT_create()` |
| 匹配策略 | 未知 | BFMatcher + knnMatch |
| Lowe 筛选 | 可能更严格（<0.5） | ratio < 0.7 |
| Cross-check | 可能开启 | 未开启 |
| RANSAC 阈值 | 未知 | 5.0 px |

### 3.3 匹配数量级差异（三级差异）

数据组仅产出 62 对匹配（43 对内点），PIS-web 产出 32,500 对匹配（882 对内点）。差异达 500 倍。

对 2448×3264 的图片，SIFT 默认提取约 30,000 个特征点。数据组 62 对匹配属于异常偏低的数值，强烈暗示其使用了与 SIFT 不同的特征检测器（如 ORB、AKAZE），或极严格的 multi-stage 筛选管线。

## 5. 结论

**PIS-web 管线内部自洽**：

```
stitch.py                        test5.py (修复后)
findHomography(src,dst)  →  H@src ≈ dst  →  RMSE ~3px  ✓
```

**数据组管线内部自洽**（推测）：

```
未知匹配代码                   test5.py (修复前/buggy)
findHomography(dst,src)  →  H@dst ≈ src  →  RMSE ~1px  ✓
```

**跨管线不可比**：两条管线在特征匹配阶段就已经分叉，产出的 H 矩阵和内点集是两套完全不同的数据。这不是分析代码（test5.py / analysis.py）的 bug，而是**上游匹配管线不同**导致的系统性差异。

**建议**: 要正确对比分析结果，应确保两套管线使用相同的匹配代码（即都走 stitch.py），仅在不同的测试图片集上比较分析指标。
