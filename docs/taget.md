【项目目标】
- 实现一个 Web 端全景图像拼接系统。用户通过网页上传多张有重叠区域的图片，后端异步处理后返回拼接后的全景图，
- 并支持历史记录查看。注意：现在只需要实现web后端即可

【技术栈】
1. 后端 Web 框架：Go + Gin框架
2. ORM 与数据库：GORM框架 + MySQL（持久化存储）
3. 缓存：Redis（用于加速状态轮询）（这个暂时可以不用实现，先直接放在内存里面吧）
4. 多语言集成：Go 通过 os/exec 调用 Python 脚本（核心算法打算使用Python）
5. 并发控制：Go 原生 Goroutine + Channel 实现异步任务队列

【核心功能】
1. 多图上传与拼接：支持一次性上传多张图片（数量不固定），后端异步拼接，前端通过 task_id 轮询获取结果。
2. 多语言承载引擎：后端不能写死 Python 调用，必须通过策略模式（Strategy Pattern）定义统一接口（AlgorithmEngine）， 
支持 Mock（模拟）、Python、C++ 三种引擎无缝切换，通过配置文件（config.yaml）动态加载。
3. 历史记录查看：提供分页查询接口，展示所有历史拼接任务（任务ID、状态、拼接耗时、特征点数、结果图预览），
4. 实现数据分析的展示，分析数据到时候会使用json文件形式

【优化逻辑】
1. 图片的传输可以使用二进制流进行传输
2. 图片可以用webp格式进行存储
3. 图片加载的优化可以使用先展示缩略图（使用imaging库），再用原图进行替换，最终务必保留原图，缩略图直接放掉，可以写到一个包里

【多语言集成逻辑】
Go 与 Python 之间通过“文件系统 + JSON”通信，禁止直接传 Base64 或复杂对象
- 输入：Go 将图片存入 ./store/uploads/{task_id}/，将目录路径传给 Python。
- 输出：Python 拼接完成后，必须在输出目录生成 meta.json（包含 status、keypoints、cost_ms、error）和 result.jpg。
- Go 的 Worker 解析 meta.json 更新数据库状态。

【工程约束与容错】
1. Go 调用 Python 必须设置 context.WithTimeout（普通模式 30 秒，超能模式 300 秒），超时或报错时优雅记录 ErrorMsg，不允许 Go 服务崩溃。
2. 后台 Worker Pool 数量固定为 20（防止同时跑太多 Python 进程导致 OOM）。
3. 上传文件大小限制为 200MB。
4. 完成/失败记录超过 40 条时自动淘汰最旧记录，同步清理磁盘文件和数据库记录。
